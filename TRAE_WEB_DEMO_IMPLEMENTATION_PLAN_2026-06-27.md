## 1. 任务目标

把当前桌面版概念复习应用，改造成一个**可公开访问、可正常体验核心功能的网页版 Demo**，用于 Trae 官方比赛初赛提交。

最终交付物不是正式商用产品，而是一个**比赛版可体验 Demo**。重点是：

- 评委能直接打开网页体验
- 核心功能完整可跑
- 页面不是静态截图，而是可交互 Demo
- 开发过程必须能在 Trae 中留痕，满足比赛规则

## 2. 比赛规则边界

已确认的比赛约束如下：

- 初赛提交物必须是“**能运行、能体验的 Demo**”
- 可以提交：

- 因此本项目的最佳提交形式是：**公开可访问的网页版链接**
- 作品可以基于已有项目做**实质性版本迭代**
- 作品开发过程必须使用 **TRAE Work 或 TRAE IDE**
- 需要保留：
  - 关键步骤截图不少于 3 张
  - 关键任务 Session ID 不少于 3 个

## 3. 本次网页化的明确成品定义

本次不要做“完整产品化重构”，而是做一个**比赛版网页 Demo**。

### 要保留的功能

- 复习卡片主流程
- 深度解读
- 自测
- 自由问答
- 概念关联导航
- 行业黑话自动发现
- 分析页星图 / 银河视图

### 不做的内容

- 不做登录系统
- 不做多用户账号体系
- 不做数据库迁移
- 不做正式 SaaS 化后台
- 不做支付、权限、团队协作
- 不做正式生产级监控系统

### 用户模型

采用：**公共访问的单 Demo 链接**

但为了避免评委互相污染数据，网页端应使用：

- **无登录**
- **每个浏览器会话独立的演示数据副本**
- **服务端统一配置 API Key**

这意味着它对外看起来是一个公共 Demo，但每个访问者使用的是自己的临时演示状态。

## 4. 技术路线与关键决策

### 4.1 总体方案

保留现有前端页面与 Python 业务逻辑，重点替换桌面桥接层：

- 保留 `frontend/index.html`
- 保留 `frontend/style.css`
- 尽量保留 `frontend/app.js` 的页面逻辑
- 保留 `backend.py` 里的核心业务与 AI 逻辑
- 保留 `main.py` 作为桌面版入口，不删除
- 新增一个网页版入口，例如：`web_app.py`

### 4.2 后端 Web 化方案

新增一个轻量 Python Web 服务，推荐直接使用 **Flask**。

原因：

- 当前后端是同步风格
- pywebview API 本来就是“方法调用型”，改成 Flask JSON API 成本最低
- 比 FastAPI 更少额外结构负担，适合比赛版快速改造

### 4.3 前端适配方案

不要重写前端页面。

应在 `frontend/app.js` 中加入一层统一 API 适配器：

- 桌面版模式：继续走 `window.pywebview.api.xxx()`
- 网页版模式：改走 `fetch('/api/...')`

目标是让同一套前端代码同时兼容：

- 桌面版
- 网页版

### 4.4 数据方案

不要让网页版直接读写用户真实使用数据。

新增一份**比赛演示种子数据**，例如：

- `data/demo_seed.json`

网页端访问流程：

1. 用户首次访问时生成匿名会话 ID
2. 服务端为该会话复制一份临时数据副本
3. 后续该会话所有复习、聊天、探索、评分都写入临时副本
4. 临时副本过期后可清理

### 4.5 配置方案

网页版不要再让前端暴露真实 API Key 输入框作为主要配置路径。

改造原则：

- DeepSeek Key 放服务端环境变量
- Brave Search Key 放服务端环境变量
- `config.json` 在网页版仅作为本地开发兜底，不作为正式比赛部署依赖

推荐环境变量名：

- `DEEPSEEK_API_KEY`
- `BRAVE_SEARCH_API_KEY`
- `DEMO_MODE=1`

### 4.6 部署形态

优先做成：

- 本地可运行
- 可部署到任意支持 Python Web 服务的平台

不要把实现绑死在某一个平台上，但交付时建议至少补齐：

- `requirements.txt`
- `web_app.py`
- 启动命令说明
- 一个简单部署说明

如果 Trae 需要进一步落地，可以在完成网页功能后再决定上哪一个平台。

## 6. 需要提供的 Web API

Trae 不要把 API 设计得太“REST 洁癖”，重点是和现有前端改造成本最低。

推荐直接按现有 pywebview 暴露方法做一一映射。

### 基础接口

- `GET /api/config-status`
- `GET /api/categories`
- `GET /api/stats`
- `GET /api/category-stats`
- `GET /api/daily-stats?days=90`
- `GET /api/review-log?limit=200`
- `GET /api/recent-cards?limit=30`

### 复习与探索

- `POST /api/due-cards`
- `POST /api/rate-card`
- `POST /api/random-explore`
- `POST /api/add-concept`
- `POST /api/delete-card`
- `POST /api/delete-category`

### 深度解读与问答

- `POST /api/deep-dive`
- `POST /api/deep-dive-payload`
- `POST /api/ask-question`
- `GET /api/chat-history?card_id=...`
- `POST /api/save-chat-history`
- `POST /api/clear-chat-history`

### 行业黑话发现

- `POST /api/discover-industry-terms`
- `POST /api/import-discovered-terms`

### 星图与关联

- `GET /api/star-map?category=...`
- `GET /api/star-detail?concept_id=...`
- `GET /api/galaxy-state`
- `GET /api/collisions`
- `GET /api/sources?concept_id=...`

说明：

- 只需要覆盖当前前端实际会调用到的接口
- `generate_relations` 如果当前前端未直接调用，可先不暴露为网页端公开接口

## 7. 实施任务拆分

下面是 Trae 应按顺序执行的任务。

### 任务 1：后端基础重构，为 Web 版做准备

目标：

- 让 `backend.py` 不再强依赖桌面本地单一路径配置
- 让它能被桌面版与网页版共同复用

必须完成：

- 让 `Backend` 支持显式传入数据目录
- 让 API Key 优先读环境变量，`config.json` 作为兜底
- 保持桌面版现有行为不被破坏
- 不删除 `main.py` 的原功能

验收标准：

- 现有桌面版仍能启动
- `tests/test_deep_dive_relations.py` 全部通过

### 任务 2：新增 Flask 网页入口

目标：

- 新增网页版主入口 `web_app.py`
- 提供静态页面和 JSON API

必须完成：

- Flask 应能启动并访问主页
- 静态资源能正常加载：
  - `index.html`
  - `app.js`
  - `style.css`
- 所有核心前端调用能找到对应 API
- 返回统一 JSON 结构

验收标准：

- 本地运行 `python web_app.py` 后可打开网页
- 首页 UI 正常显示，不是白屏

### 任务 3：前端 API 调用层改造

目标：

- 把 `window.pywebview.api.xxx()` 改造成统一调用层

推荐做法：

- 在 `frontend/app.js` 顶部新增一个 API 适配对象
- 对外暴露统一方法名
- 内部自动判断运行环境：
  - 有 `window.pywebview.api` 就走桌面版
  - 否则走 `fetch`

必须完成：

- 不改页面业务流程
- 只替换调用通道
- 保持函数名和调用语义尽量不变

验收标准：

- 桌面版仍可用
- 网页版能正常加载和调用后端

### 任务 4：网页端比赛版 UI 收口

目标：

- 把桌面思维下的配置方式收成比赛版网页可用状态

必须完成：

- 网页版不要要求评委自己填写 API Key
- 设置页在网页端应改成：
  - 展示服务端是否已配置
  - 或弱化成只读提示
- 页面文案中不出现“请用户本地填写 key 才能用”的核心阻塞体验

可选优化：

- 在网页首页补一行“比赛演示版”提示
- 在异常提示中把“本地配置”改成“服务暂不可用，请稍后重试”

验收标准：

- 评委首次打开网页，不需要任何本地配置即可体验核心功能

### 任务 5：匿名会话演示数据机制

目标：

- 避免多个评委共享同一份状态

必须完成：

- 服务端为新访问者生成匿名会话
- 每个会话有自己的演示数据副本

建议实现：

- 用 cookie 存匿名 session id
- 会话首次访问时，从 `data/demo_seed.json` 复制到临时目录
- 临时目录可放在系统 temp 或项目可清理目录

验收标准：
